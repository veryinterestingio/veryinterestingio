    TurboFucked A:

    Exploiting video games isn't all that different from exploiting other types of software. Generally, you try to do something unexpected, excessive, or just plain quickly, which causes the software to behave weirdly, then you keeping doing that action slightly differently, until you achieve the desired result.

    It helps to understand how games (and software in general) work on a fundamental level. For example, collision detection is used to determine if two or more bodies are "touching" in the game world. If you understand the various algorithms used to perform collision detection as well as the fact that numbers with decimal points have weird behavior on computers, then you'll realize some common "tests" you can perform in maps that may cause you to go through walls or something.

    Things like item duplication are mostly luck. A developer made an assumption about items that the player ended up breaking.

    For some games, you can hook up what's called a "debugger" to the game application and explore how the game was programmed. You can then do things like stop the application and walk through execution line by line, inspecting how values in memory change as various actions are performed. This is called a break point.

    Generally, a hacker would do a bunch of crazy stuff in the game until they find a glitch, freeze, crash, slowdown, or maybe just an error message. When that happens, they will fire up a debugger and attached it to the game. They will then put break points in the application and various places and trigger glitch, freeze, crash, etc, stepping through the code to see what is causing the bug.

    Once they see what the bug is causing the crash, the hacker can then tailor their actions in such a way that they can trigger the bug, but instead of crashing, it will cause the program to do what they want.

    So in the example of item duplication, let's say the hacker is doing a bunch of weird stuff in the inventory screen, like dragging items where they shouldn't be, etc, until they find a glitch. They then hook up the debugger and notice that oh hey, the code remembers the slot that I picked up an item from, and it blindly does a stackSize++; when the item is dropped somewhere it doesn't belong, then clear the item from the cursor. But if they put the cursor over the avatar's head and release the mouse button, the stack size incrementing code triggers, but the clear item from cursor doesn't. So that can be repeated to get infinite items.

    TL;DR: Knowing how computers / software work, plus practice.

    Gaboik A:

    Sometimes, other than reverse engineering the game's executable files, people can also reverse engineer the application protocol used by the game client and server to communicate over the internet.

    What you do is eavesdrop on everything your computer and the game server are saying to each other. And if the game is poorly coded, you can see thing like the client saying to the server "give me 1 more of this item" and the server blindly agrees. You are then free to replay (resend) the same request over and over.

    It's become very unlikely that games are coded like that nowadays but it's been seen in the past, I believe there was something pretty similar to this with Guild Wars a while back.
