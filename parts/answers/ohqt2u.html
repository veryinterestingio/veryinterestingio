Suarezlasky A:

Main issue is trusting a security configuration to the user. Of the top of my head I can think of some problems like:

-Trivial questions: there would be a lot of people that would just put something like "what's your birthday?".

-Misuse of custom field: people could just put "your password is 1234ravioli".

-Not necessarily better: the fact that you can put "what is the middle name of the cat of my neighbor from childhood?" Doesn't really means someone who is super skilled at social engineering can get to that answer somehow.

Also there is a technical problem: you have to store those questions somewhere. If you have let's say 10 default security questions labeled 1-10, then each user will need just a number to know which security question belongs to them (1 byte at most), but for every user to have a unique custom security question you'll have to give them around 50 characters of storage (50 bytes), it might not sound like much, but it really depends on how big is your user base.

There is probably more considerations to be had, but I think what's written above is enough to make a case.
