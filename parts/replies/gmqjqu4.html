    In addition to locating executable components, there are also many programs that have internal settings (often stored in the windows registry) that refer to specific locations on the filesystem for locating data/resources. These paths are recorded during the initial installation process, and the program may malfunction when moved elsewhere.

    Other related issues include:

    Convenience of pre-installed shortcuts that set the working folder appropriately, especially when it is not the same as the folder containing the executable.

    Questionable (read:terrible) programming practices such as hard-coded paths (I am very guilty of this when writing small tools), assumptions such as "the Windows folder must ALWAYS be on drive C:", or assuming that the program installation will always be on the same drive as some other resource being sought (inappropriate use of relative paths, etc.).
