  • purebuu

    The why is, that line by line was invented first because its a really simple idea. All images in the web loaded like this. But internet connections were slow back then, images took seconds/minutes to load.

    So some smart person came up with the idea of progressive images, the image is stored not line by line but staggered, so the first pixels read will form a very low resolution version of the whole image. This allows the first few bytes streamed to people computers to make up the whole image even if blurry/pixelated (the blur effect is done on the client machine not in the image). Greatly speeding up how quickly a person can 'see' what the picture is while its still downloading.

    It is still left up to the image creator if they want to create line by line images or progressive ones. If you take a picture of a bird, progressive might make sense, but scan a page of text line by line would be better.

  • bloc97

    I haven't seen people talk about code complexity and compression.

    Usually top to bottom streaming is easier to code, thus widely supported. Older file formats usually always support this. Also it is easier to compress the data if it is streamed from start to finish because you can compress images better by packing up nearby similar pixels into chunks. Real life images always have patches of pixels that have the same or similar colors. If you were to send every nth pixel, it would be difficult to compress since the pixels are less correlated.
