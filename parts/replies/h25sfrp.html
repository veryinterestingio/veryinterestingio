    I liked the detail about serialization =). BTW what are the differences (conceptual and/or technical) between an API and any other interface you can create for your code/program/library?

    • lobax

      If you have an interface for other programmers (or even just yourself) that you expose for use between modules and/or programs, then that is an API. This things can be fully internal, like a library built by one team that is used by other teams in the same company.

      The key thing is that the interface the API provides can be left unchanged while the code behind it is abstracted away and can be continuously updated without those using the api having to care.

    • BattleAnus

      Imagine a codebase that has classes to handle getting the time and date in different timezones (e.g. getDateTimeEST() or getDateTimePST(), etc.).

      It could have an API that sits as an extra layer between the user and those classes and picks the correct class based on the location of the user, so the user simply makes a request to example.server/api/getDateTime and it returns the time based on a specific class. This would be an example of an API.

      You could also imagine that if the user had access to the code for the classes themselves, he might right some code that simply imports the class he wants and makes the call directly: import getDateTimeEST; print getDateTimeEST();. This would be one way of interfacing with code that isn't an API.

      Another (admittedly contrived) way of interfacing would be for the datetime app to update a publically-accessible text file with the times of each timezone, and so the user would simply read from that text file to get the time, possibly doing some filtering to get the datetime they want. Alternatively this could be stored in memory instead of on a hard disk, as long as the user's program had access to that bit of memory.

      These are some ways that a user could utilize the code of the app that aren't considered an API
