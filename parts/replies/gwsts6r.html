    What if we use cookies instead? Will that have a significant effect on the reset timing?

    • ltburch

      We stray further from ELI5, but there are several strategies for handling this. You can use cookies or local storage but the rub there is that it is insecure - the user can monkey with their browser to send whatever cookies they want so it isn't really trusted (not that people don't sometimes implement it this way).

      However all is not lost! The trick is you use cookies or local storage but you encrypt it before you send it to the users browser. This essentially makes the data in the cookie "opaque", it is also essentially immutable as you can't monkey with it since if you do it will not properly decrypt on the server. Often timestamps and browser IP are embedded in this to avoid replay attacks (other users sending your encrypted data)

      There are some other tricks as well but it isn't that relevant to a general understanding.

    Why do we want to use sessions anyway?

    • ltburch

      Generally there are three answers to this:

      1) The session data is large and potentially dynamic and we don't want to have to be re-sending that to the browser every time.

      2) We don't want the browser to actually have direct access to the data, there may be secrets in the data like encryption keys we would prefer to keep on the server.

      3) We want to control the lifetime/validity of the data on the server (this is particularly relevant for authentication)
