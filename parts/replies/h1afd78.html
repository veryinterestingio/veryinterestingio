      • xevizero

        I think it does in some cases. For example, extracting all files from an archive (which is inside the installer) in a specific preallocated position on disk is pretty fast. Then the installation requires you to move a lot of small files one by one in their correct location (different folders maybe even on a different drive), making changes to the registry, checking license information, verifying data integrity and a lot of other stuff that relies on the speed of your drive in random I/O and on the responsiveness of your OS or even internet connection latency. So if the progress bar was made to be 80% unpacking and copying files and 20% "setting things up" (which means a lot of different things), it could very well be that on slower PCs the last 20% is very slow, while on other PCs you could see the inverse being true, so the first part of the installation happens than in an instant the last 20% is done.

      • chaossabre

        Decompress data into a temporary staging location, then copy to intended destination, then delete data from staging location. Decompression is CPU-heavy so "90%" and file operations are I/O bound and thus CPU-light "10%". File operations are also heavily dependent on storage system performance: HDD vs Flash vs NVMe storage which covers the high variability OP is asking about.

        It's not a perfect explanation but it is grounded in reality.
